# M28 쿠폰 사용 및 주문 할인 적용 설계

## 목표와 범위

발급된 구매자 쿠폰을 단일 상품 주문에 선택 적용하고, 프로모션 할인 뒤의 쿠폰 할인 및 최종 결제금액을 주문에 불변 스냅샷으로 보관합니다. 쿠폰은 주문 생성 때 최대 30분간 예약하고 결제 승인 때만 사용 처리합니다.

이번 마일스톤은 쿠폰 발급 한도(M27)를 변경하지 않습니다. 복수 상품 주문, 복수 쿠폰 중첩, 쿠폰 재발급, 취소·환불에 따른 쿠폰 복구도 범위 밖입니다.

## 확정 정책

- 구매자는 적용 가능 쿠폰 중 0개 또는 1개를 직접 선택합니다. 자동 적용하지 않습니다.
- 할인 순서는 `정가 - 프로모션 할인 - 쿠폰 할인`입니다. 쿠폰의 최소 구매금액 판단도 프로모션 할인 후 금액을 사용합니다.
- `stackable=true` 쿠폰만 활성 프로모션과 함께 적용할 수 있습니다. 프로모션이 없으면 `stackable` 값과 무관하게 적용할 수 있습니다.
- 정액 쿠폰은 프로모션 후 금액을 넘겨 할인할 수 없습니다. 정률 쿠폰은 `(프로모션 후 금액 × 할인율) / 100`의 소수점 이하를 버리고, 그 뒤 `maxDiscountAmount` 상한을 적용합니다. 최종 결제금액의 하한은 0원입니다.
- 0원 주문은 외부 PG를 호출하지 않고 내부 결제 승인으로 완료합니다.
- 쿠폰의 사용 가능 여부는 발급 시점에 저장한 `MemberCoupon` 스냅샷과 `validUntil`로 판단합니다. 발급 뒤 캠페인이 일시중지되거나 종료되어도 유효기간 안이면 사용할 수 있습니다.
- 예약 시점에 유효한 쿠폰은 이후 `validUntil`이 지나더라도 예약 만료시각 전까지 결제를 승인할 수 있습니다.
- 결제 승인 뒤 쿠폰은 `USED`로 확정되며, 이후 주문 취소 또는 환불이 승인되어도 복구하지 않습니다.
- 결제 승인 실패, 구매자 주문 취소, 주문 생성 트랜잭션 실패에는 쿠폰을 사용 처리하지 않고 예약만 해제합니다.
- 결제되지 않은 예약은 주문 생성 30분 후 자동 만료됩니다. 만료 시 주문을 취소하고 상품·재고 예약을 기존 주문 취소 흐름으로 복구합니다.

## 데이터 모델

### `coupon_reservations`

새 테이블은 쿠폰 사용 예약의 이력과 상태를 책임집니다.

| 열 | 의미 |
| --- | --- |
| `id` | 예약 식별자 |
| `member_coupon_id` | 예약된 구매자 쿠폰 |
| `order_id` | 예약을 만든 주문 |
| `status` | `RESERVED`, `CONSUMED`, `RELEASED`, `EXPIRED` |
| `reserved_at` | 예약 생성 시각 |
| `expires_at` | 예약 만료 시각(생성 후 30분) |
| `consumed_at` | 결제 승인 확정 시각, 그 외 `null` |
| `released_at` | 해제·만료 처리 시각, 그 외 `null` |

`member_coupon_id`에 대해 `status = 'RESERVED'`인 행은 하나만 허용하는 PostgreSQL 부분 유니크 인덱스를 둡니다. `order_id`도 유일하게 해 주문당 쿠폰 예약을 하나로 제한합니다. 이력 행은 지우지 않습니다.

`member_coupons.status`는 `ISSUED`와 `USED`를 계속 책임집니다. 예약은 이 상태를 바꾸지 않습니다. 따라서 주문 진행 중인 쿠폰은 활성 예약 존재 여부로 선택 불가가 됩니다.

### 주문 스냅샷

`orders`에 다음을 추가합니다.

- `member_coupon_id`: 선택된 쿠폰의 ID, 미선택 시 `null`
- `coupon_discount_amount`: 쿠폰 할인액, 기본값 `0`

기존 `list_price`, `promotion_campaign_id`, `promotion_discount_amount`, `final_price`와 함께 주문 시점 가격을 재현합니다. 과거 주문은 `member_coupon_id = null`, `coupon_discount_amount = 0`으로 호환합니다.

## 적용 가능 여부와 할인 계산

주문 생성은 선택 쿠폰을 잠근 뒤 다음을 검증합니다.

1. 쿠폰이 요청 구매자 소유이고 상태가 `ISSUED`이며, 활성 예약이 없습니다.
2. 예약 생성 시각이 `validUntil`보다 이릅니다.
3. `ALL_PRODUCTS`이거나 `SELECTED_PRODUCTS` 대상에 주문 상품이 포함됩니다.
4. 프로모션이 실제 적용된 주문에서 쿠폰이 `stackable=false`이면 거절합니다.
5. 프로모션 후 금액이 `minimumPurchaseAmount` 이상입니다.

쿠폰은 발급 시 복사된 적용 범위·할인 종류·할인값·최대 할인액·최소 구매금액·중복 가능 여부만 사용합니다. 가변적인 캠페인 상태나 캠페인의 이후 수정값을 다시 읽지 않습니다.

## 트랜잭션과 상태 흐름

### 주문 생성

선택 쿠폰이 있으면 주문 생성 트랜잭션 안에서 `MemberCoupon`을 비관적 잠금으로 읽고 검증합니다. 프로모션 가격을 계산한 뒤 쿠폰 할인액과 최종 가격을 계산해 주문을 저장하고 `RESERVED` 예약을 생성합니다. 부분 유니크 인덱스 충돌은 이미 예약된 쿠폰 오류로 변환합니다.

쿠폰이 없으면 기존 주문 생성 흐름과 동일하게 주문만 저장합니다.

### 결제 승인

결제 승인 트랜잭션은 주문, 활성 예약, 쿠폰을 잠급니다. 예약이 `RESERVED`이고 `expiresAt` 전이면 외부 PG 승인 성공 뒤 같은 트랜잭션에서 예약을 `CONSUMED`로, 쿠폰을 `USED`로, 주문을 `PAID`로 전이합니다. 이 전이는 멱등적으로 처리합니다.

외부 PG 실패, 승인 전에 발생한 도메인 오류, 구매자 주문 취소는 활성 예약을 `RELEASED`로 전이합니다. 쿠폰은 `ISSUED`로 남습니다. 기존 재고·단품 상품 예약 해제 흐름도 유지합니다.

최종 결제금액이 0원이면 PG를 호출하지 않고 동일한 내부 승인 트랜잭션을 실행하여 내부 결제 레코드, 주문 `PAID`, 예약 `CONSUMED`, 쿠폰 `USED`를 함께 확정합니다.

### 예약 만료

주기 작업은 `expires_at <= now`인 `RESERVED` 예약을 잠가 처리합니다. 해당 주문이 `CREATED`면 기존 취소 메서드로 상품·재고 예약을 복구하고, 예약을 `EXPIRED`로 전이합니다. 이미 승인·취소·만료된 경우에는 상태를 재변경하지 않아 반복 실행에도 안전합니다.

### 취소와 환불

결제 전 취소는 예약을 해제합니다. 결제 후 취소·환불은 예약이나 쿠폰을 되돌리지 않습니다. 이미 `CONSUMED`인 예약과 `USED` 쿠폰, 주문의 가격 스냅샷은 감사·정산 목적으로 유지합니다.

## API와 화면

- 구매자용 적용 가능 쿠폰 조회 API는 상품 ID를 받아 쿠폰 ID, 표시 정보, 계산된 쿠폰 할인액, 예상 최종가격을 반환합니다. 적용 불가 쿠폰은 이 목록에서 제외합니다.
- 주문 생성 요청의 선택값 `memberCouponId`는 `null`을 허용합니다. 서버는 조회 응답을 신뢰하지 않고 생성 시점에 다시 검증·계산합니다.
- 주문, 결제, 환불, 정산 관련 응답은 `memberCouponId`와 `couponDiscountAmount`를 반환합니다. 기존 소비자는 새 필드를 무시해도 됩니다.
- 구매자 주문 화면은 적용 가능 쿠폰을 라디오/선택 목록으로 표시하고, 선택 해제와 함께 예상 결제금액을 갱신합니다.

## 오류와 검증

오류는 쿠폰 소유자 불일치, 이미 사용됨, 활성 예약 존재, 예약 만료, 쿠폰 만료, 적용 상품 불일치, 최소 구매금액 미달, 프로모션 중복 불가를 구분합니다.

테스트는 다음을 포함합니다.

- 정액·정률 할인, 정률 소수점 버림, 최대 할인액, 최종 0원
- 범위·최소 구매금액·중복 가능 여부·발급 후 캠페인 중지/종료
- 같은 쿠폰의 동시 주문 한 건만 예약 성공
- 결제 성공 소비, 결제 실패/결제 전 취소 해제, 결제 후 취소·환불에도 `USED` 유지
- 30분 예약 만료의 주문·재고 복구와 반복 실행 안전성
- 0원 주문의 내부 승인과 기존 결제·정산 조회 호환
- 쿠폰 미선택 및 마이그레이션 이전 주문의 호환

새 JUnit `@Test` 메서드명은 한국어와 밑줄을 사용합니다.
